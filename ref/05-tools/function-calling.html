Function Calling with the Gemini API

Function calling allows the Gemini model to determine when to invoke specific functions and provide necessary parameters, acting as a bridge between natural language and real-world actions.

Core Use Cases

  • Augmenting knowledge via external sources
  • Extending model capabilities with tools
  • Taking real-world actions through APIs

How Function Calling Works (Four-Step Flow)

  1. Define function declarations describing name, parameters, and purpose
  2. Call API with user prompt and function declarations — model responds with structured JSON
  3. Execute function in your application code (the model does not execute it)
  4. Return results to the model for generating a user-friendly response

Function Declaration Structure

{
  "name": "function_name",
  "description": "Clear explanation of purpose with examples",
  "parameters": {
    "type": "object",
    "properties": {
      "param_name": {
        "type": "string",
        "description": "Parameter purpose",
        "enum": ["optional", "fixed", "values"]
      }
    },
    "required": ["mandatory_params"]
  }
}

Key declaration fields:

  • name: Unique identifier (use underscores or camelCase; no spaces/periods/dashes)
  • description: Detailed explanation with examples and format hints (e.g., "San Francisco, CA" or "95616")
  • parameters.type: Data structure type
  • parameters.properties: Individual parameter definitions
  • parameters.required: Mandatory parameter array
  • enum: Fixed value constraints (improves accuracy — use instead of just describing limited values)

Python Code Example

from google import genai
from google.genai import types

client = genai.Client()

def get_weather(location: str) -> str:
    # Your actual implementation here
    return f"The weather in {location} is sunny, 25°C"

tools = [
    types.Tool(function_declarations=[{
        "name": "get_weather",
        "description": "Get current weather for a location",
        "parameters": {
            "type": "object",
            "properties": {
                "location": {
                    "type": "string",
                    "description": "City and state, e.g. 'San Francisco, CA'"
                }
            },
            "required": ["location"]
        }
    }])
]

response = client.models.generate_content(
    model="gemini-3.5-flash",
    contents="What's the weather in Tokyo?",
    config=types.GenerateContentConfig(tools=tools)
)

# Check if model wants to call a function
if response.candidates[0].content.parts[0].function_call:
    fc = response.candidates[0].content.parts[0].function_call
    # fc.name = "get_weather"
    # fc.args = {"location": "Tokyo"}
    # fc.id = unique identifier (Gemini 3+)

    result = get_weather(**fc.args)

    # Return result to the model (include the id for Gemini 3)
    response2 = client.models.generate_content(
        model="gemini-3.5-flash",
        contents=[
            {"role": "user", "parts": [{"text": "What's the weather in Tokyo?"}]},
            {"role": "model", "parts": [{"function_call": {"name": fc.name, "args": fc.args, "id": fc.id}}]},
            {"role": "user", "parts": [{"function_response": {"name": fc.name, "response": {"result": result}, "id": fc.id}}]}
        ]
    )
    print(response2.text)

Unique IDs (Gemini 3+)

Gemini 3 models always return a unique id with every functionCall. You must include this exact ID in your functionResponse for accurate mapping. This is critical for parallel function calling.

Function Calling Modes

ModeBehavior
AUTODefault; model chooses between function calls or natural text response
ANYModel always predicts a function call; ensures schema adherence
VALIDATEDDefault for tool combinations; constrains to valid outputs
NONEProhibits function calling entirely
tool_config = types.ToolConfig(
    function_calling_config=types.FunctionCallingConfig(
        mode="ANY",
        allowed_function_names=["function1", "function2"]
    )
)

allowedFunctionNames (toolConfig.functionCallingConfig)

The allowedFunctionNames parameter restricts which functions the model may select when making function calls — it acts as a filter over the available function declarations. When omitted, the model can choose from all available declarations; when provided, the model's choices are narrowed to only the listed functions.

  • ANY mode: if allowedFunctionNames is provided as a list, the model can only choose from the functions in that list.
  • VALIDATED mode: if allowedFunctionNames is provided, the model picks from the set of allowed functions.

This is most commonly used with mode ANY to force a call to one of a specific subset of functions. (In the Python SDK, the field is allowed_function_names, as shown above.)

Parallel Function Calling

Enables multiple simultaneous function executions when dependencies don't exist. The API maps results to calls using the id field.

Key advantage: "You don't need to return the function_result objects in the same order that the function_call objects were received."

Compositional (Sequential) Function Calling

Allows chaining multiple functions sequentially. Example: first calling get_current_location(), then get_weather() using the location result. The Live API natively supports compositional calling without manual execution loops.

Automatic Function Calling (Python SDK Only)

Pass Python functions directly; the SDK automatically converts them to declarations, detects calls, executes functions, and sends results back.

Requirements: type hints and Google-style docstrings.

Allowed parameter types: int, float, bool, str, list, and Pydantic BaseModel

def get_weather(location: str) -> str:
    """Get current weather for a location.

    Args:
        location: City and state, e.g. 'San Francisco, CA'
    Returns:
        Weather description string
    """
    return f"Sunny, 25°C in {location}"

# SDK automatically handles function calling
response = client.models.generate_content(
    model="gemini-3.5-flash",
    contents="What's the weather in Tokyo?",
    config=types.GenerateContentConfig(tools=[get_weather])
)

# Disable automatic calling if needed:
# automatic_function_calling=types.AutomaticFunctionCallingConfig(disable=True)

Multimodal Function Responses (Gemini 3)

Supports nested multimodal content within functionResponse:

  • Image formats: PNG, JPEG, WebP
  • Document formats: PDF, plain text

Reference multimodal parts using {"$ref": "displayName"} in the structured response field.

Model Context Protocol (MCP)

Experimental feature in Python and JavaScript SDKs:

  • Python: automatic tool calling via ClientSession
  • JavaScript: via mcpToTool() function
  • Supports tools only (not resources or prompts)
  • Disable automatic calling with automatic_function_calling config
  • Breaking changes possible in future releases

Multi-Tool Use (Tool Combination)

Gemini 3 models can combine built-in tools with custom function declarations. Set:

include_server_side_tool_invocations=True

when mixing Google Search with custom functions.

Thought Signatures (Gemini 3 with Thinking)

For models with thinking enabled, signatures maintain context across turns. The SDKs handle this automatically. Manual manipulation requires:

  • Always sending thought_signature back in the original Part
  • Including exact function call id in function_response
  • Never merging signature-containing parts

Best Practices

  1. Descriptions: Be extremely clear and specific; include examples and format hints
  2. Naming: Descriptive names without spaces, periods, or dashes
  3. Strong typing: Use specific types and enum for fixed value sets
  4. Tool selection: Limit to 10–20 active tools; use dynamic selection for large tool sets
  5. Prompt engineering: Provide role context, usage instructions, encourage clarification
  6. Temperature: Use low values (0.0) for deterministic calls; Gemini 3 recommends keeping default 1.0
  7. Validation: Confirm high-consequence calls before execution
  8. Error handling: Return informative error messages in function responses
  9. Security: Use proper authentication; avoid exposing sensitive data in declarations
  10. Check finish reason: Always validate finishReason in responses
  11. Token limits: Function descriptions count toward input token limits

Notes and Limitations

  • Uses OpenAPI schema subset only (not full specification)
  • Very large or deeply nested schemas may be rejected in ANY mode
  • Don't assume functionCall is last in parts array when mixing tools
  • Automatic function calling is a Python SDK feature only
  • Python parameter type support is limited to basic types and Pydantic BaseModel
