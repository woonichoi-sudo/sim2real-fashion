Safety and Factuality Guidance for Gemini API

"Generative artificial intelligence models are powerful tools, but they are not without their limitations. Their versatility and applicability can sometimes lead to unexpected outputs, such as outputs that are inaccurate, biased, or offensive."

Key requirement: "Post-processing, and rigorous manual evaluation are essential to limit the risk of harm from such outputs."

Applicable Policies

  • Use governed by Generative AI Prohibited Use Policy and Gemini API terms of service
  • Built-in content filtering with adjustable safety settings across 4 dimensions
  • Optional grounding with Google Search for improved factuality

Section 1: Understanding Safety Risks

Definition: "Safety is being defined as the ability of an LLM to avoid causing harm to its users, for example, by generating toxic language or content that promotes stereotypes."

Key Responsibility: "As the application owner, you are responsible for knowing your users and the potential harms your application may cause, and ensuring that your application uses LLMs safely and responsibly."

Risk Assessment Process

  • Evaluate likelihood of harm occurrence
  • Determine seriousness level
  • Identify mitigation steps
  • Example contrast: Factual essay generation requires more misinformation vigilance than fictional entertainment apps

Advanced Tips - User Research

  • Consult diverse prospective user groups about application purpose
  • Research state-of-the-art studies in application domain
  • Observe similar app usage patterns
  • Conduct user studies, surveys, or informal interviews

Resource References

  • AI Risk Management Framework (NIST)
  • DeepMind publication on "ethical and social risks of harm from language models"

Section 2: Mitigation Strategies

Approach 1: Tuning Model Output

  • Make outputs more predictable and consistent
  • Better reflect acceptable context-specific results

Approach 2: Facilitate Safer Inputs

  • Experiment with input prompts for safety optimization
  • Restrict users to drop-down prompt selections
  • Offer pop-up suggestions with tested descriptive phrases

Approach 3: Input/Output Filtering

  • Block unsafe inputs and filter outputs before user display
  • Simple blocklists for unsafe words/phrases
  • Manual human reviewer alterations/blocks

Warning: "Automatically blocking based on a static list can have unintended results such as targeting a particular group that commonly uses vocabulary in the blocklist."

Approach 4: Trained Classifiers

  • Label prompts with potential harms or adversarial signals
  • Deploy strategies based on detected harm type
  • Block overtly adversarial/abusive input with pre-scripted responses

Advanced Options for Harmful Outputs:

  • Provide error message or pre-scripted output
  • Retry prompt (same prompt may generate different safe outputs)

Approach 5: Deliberate Misuse Safeguards

  • Assign unique user IDs
  • Impose query volume limits per time period
  • Protect against prompt injection attacks

Definition: "Prompt injection, much like SQL injection, is a way for malicious users to design an input prompt that manipulates the output of the model."

Approach 6: Lower-Risk Functionality

  • Narrow scope tasks (keyword extraction)
  • Increase human oversight (reviewed short-form content)
  • Examples: Instead of full email generation, offer outline expansion or alternative phrasings

Approach 7: Adjustable Safety Settings

  • Modify safety configuration during prototyping
  • Adjust across filter categories to restrict/allow content types
  • Reference: Safety settings guide

Approach 8: Enable Grounding

  • Use Grounding with Google Search to reduce hallucinations
  • "The Grounding with Google Search feature connects the Gemini model to real-time web content and works with all available languages. This allows Gemini to provide more accurate answers and cite verifiable sources beyond the models knowledge cutoff."

Section 3: Safety Testing

Context Matters: Risk severity varies by application and implementation. A haiku generator poses different risks than legal document summarization tools. Human expert review prior to action reduces harmful output likelihood.

Test Type 1: Safety Benchmarking

  • Design safety metrics reflecting potential application harms
  • Test performance using evaluation datasets
  • Establish minimum acceptable safety metric levels before testing
  • Gather evaluation datasets based on priority metrics

Advanced Tips:

  • Don't rely solely on "off-the-shelf" approaches
  • Build custom testing datasets using human raters for context fit
  • Address metric trade-offs when improvements in one area hurt another
  • Consider worst-case performance over average performance

Test Type 2: Adversarial Testing

  • Proactively attempt to break the application
  • Identify weakness points for remediation
  • Requires evaluators with application domain expertise

Malicious Input Definition: "An input may be malicious when the input is clearly designed to produce an unsafe or harmful output-- for example, asking a text generation model to generate a hateful rant about a particular religion."

Inadvertently Harmful Input Definition: "An input is inadvertently harmful when the input itself may be innocuous, but produces harmful output -- for example, asking a text generation model to describe a person of a particular ethnicity and receiving a racist output."

Test Data Selection Strategy

  • Select data most likely to elicit problematic output
  • Probe all possible harm types
  • Include rare/unusual examples and relevant edge-cases
  • Incorporate diversity across sentence dimensions (structure, meaning, length)
  • Reference: Google's Responsible AI practices in fairness

Advanced Tip: "Use automated testing instead of the traditional method of enlisting people in 'red teams' to try and break your application. In automated testing, the 'red team' is another language model that finds input text that elicit harmful outputs from the model being tested."

Note: "LLMs are known to sometimes produce different outputs for the same input prompt. Multiple rounds of testing may be needed to catch more of the problematic outputs."

Section 4: Ongoing Monitoring

Planning Principle: "No matter how much you test and mitigate, you can never guarantee perfection, so plan upfront how you'll spot and deal with problems that arise."

Monitoring Approaches

  • Establish monitored channels for user feedback (thumbs up/down ratings)
  • Run user studies with diverse participant mix
  • Proactively solicit feedback from users with different usage patterns

Advanced Tip: "When users give feedback to AI products, it can greatly improve the AI performance and the user experience over time by, for example, helping you choose better examples for prompt tuning."

Resource: Google's People and AI guidebook - "Feedback and Control" chapter

Recommended Implementation Cycle

The iterative model implementation cycle with phases:

  1. Understanding safety risks
  2. Considering adjustments
  3. Performing safety testing
  4. Soliciting user feedback and monitoring

"The adjustment and testing phases should be iterative until you reach performance appropriate for your application."
