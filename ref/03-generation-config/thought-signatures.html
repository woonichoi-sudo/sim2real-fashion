Thought Signatures - Gemini API Documentation

Overview

Thought signatures are encrypted representations of a model's internal reasoning, preserving context across multi-step interactions. They're automatically handled by official SDKs but require manual management with REST APIs.

Key Concepts

Turn vs. Step:

  • A "turn" = complete user-model exchange
  • A "step" = individual operation within a turn (e.g., single function call)

Function Call Signatures

Single/Sequential Calls

  • First functionCall part contains thoughtSignature
  • Must return signature in exact location received
  • Gemini 3 requirement: Omitting signatures causes 400 error

Parallel Calls

  • Only first functionCall part includes signature
  • Subsequent calls in same response lack signatures
  • Must preserve first signature when returning

Validation Rules

"The API goes back in the history (newest to oldest) to find the most recent User message that contains standard content" to establish current turn scope.

Requirements for current turn:

  • First functionCall in each step must include thoughtSignature
  • All previous turns don't require validation
  • Omission results in: "Function call <Function Call> in the <index> content block is missing a thought_signature"

Non-Function Call Signatures

Text/in-context reasoning parts may contain signatures:

  • Recommended to return for reasoning quality
  • Not strictly enforced - omission won't cause errors
  • Helps maintain complex instruction following

Token Management (Gemini 3.5 Flash+)

"Beginning with Gemini 3.5 Flash, the model uses the reasoning context from all previous turns when thought signatures are present"

Cost consideration: Preserving signatures increases input tokens in subsequent turns.

OpenAI Compatibility Format

Signatures stored in extra_content.google.thought_signature within tool_calls array objects.

Common Errors and Workarounds

Missing signatures: "Failure to include signatures will result in a 400 error"

Dummy signatures for legacy imports:

  • "context_engineering_is_the_way_to_go"
  • "skip_thought_signature_validator"

Parallel function interleaving issue: Expected format "FC1+sig, FC2, FR1, FR2" not "FC1+sig, FR1, FC2, FR2"

Streaming without function calls: Check for empty text parts containing signatures; parse until finish_reason appears.

SDK Handling

The documentation emphasizes that "thought signatures are handled automatically when you use the official Google Gen AI SDKs" and only require manual handling with REST API implementations or when manually constructing conversation history.

Critical Rules for Function Calling (Gemini 3)

  1. Mandatory for current turn: You must include thought signatures for all function calls within the active turn, or requests will fail with a 400 error
  2. Signature placement: In parallel function calls, only the first functionCall part receives a signature; subsequent parallel calls do not
  3. Sequential calls: Each step in a multi-step sequence requires its signature preserved exactly as received
